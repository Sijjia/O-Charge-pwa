# RBAC — роли и границы ответственности

> Цель: четкие роли и права доступа для клиентов и партнеров. Следуем RULES.md.

## Роли

- client: доступ к клиентскому приложению (зарядка, баланс, история)
- operator: управление станциями в пределах своей организации/локаций
- admin: полный доступ в рамках организации (локации, станции, пользователи организации)
- superadmin: глобальный доступ (все организации, системные настройки)
- auditor (read-only): просмотр аналитики/логов без изменений

## Границы доступа (примеры)

- client:
  - читать публичные локации/станции, запускать/останавливать зарядку для себя
  - видеть собственный баланс/платежи/историю
- operator:
  - CRUD станций своей организации, видеть сессии и инциденты своей сети
  - без доступа к настройкам биллинга организации
- admin:
  - все права operator + управление локациями/тарифами в организации, текущее назначение ролей
- superadmin:
  - управление организациями, глобальные настройки, аудит, аварийные операции
- auditor:
  - только чтение: отчеты, логи, инциденты, без мутаций

## Реализация (frontend)

- Флаг роли приходит из backend профиля пользователя
- Route guards/feature gates на основе ролей
- Скрытие UI-элементов вне зоны доступа
- Единый источник ролей — `useUnifiedAuthStore.user.roles`

## Реализация (backend, рекомендации)

- RLS (Row-Level Security) для таблиц станций/локаций/организаций
- Политики на основе `org_id` и `role` из JWT/сессии
- Аудит изменений (кто/что/когда)
