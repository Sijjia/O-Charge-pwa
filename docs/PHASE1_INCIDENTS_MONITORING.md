# Фаза 1 — Incidents & Monitoring (план внедрения)

> Цель: прозрачность состояния сети и быстрая реакция на сбои. Следуем `RULES.md`: анализ → реализация.

## Обзор (анализ)

- Нужны каналы уведомлений для админов по инцидентам (офлайн/ошибки/платежи).
- Требуются метрики SLO/SLA и дешборды (uptime локаций/станций/коннекторов).
- Нужен аудит действий партнера (CRUD, тарифы, операции).
- Отчёты об инцидентах и экспорт.

## Чеклист внедрения

- [ ] Каналы уведомлений для админов:
  - [ ] Email (SMTP)
  - [ ] Telegram/Slack (бот/вебхуки)
  - [ ] Правила приоритета/эскалации
- [ ] Правила инцидентов:
  - [ ] Станция офлайн/онлайн
  - [ ] Ошибка коннектора/сессии
  - [ ] Сбои платежей/провайдеров
  - [ ] Пороговые события (низкий uptime)
- [ ] Метрики и дешборды:
  - [ ] Uptime локаций/станций/коннекторов
  - [ ] MTBF/MTTR
  - [ ] Трафик/ошибки API
- [ ] Аудит:
  - [ ] Логирование действий (кто/что/когда)
  - [ ] Фильтры/поиск/экспорт
- [ ] Отчеты:
  - [ ] Ежедневные/недельные суммарные отчёты инцидентов
  - [ ] Экспорт CSV/PDF

## Критерии готовности

- [ ] Инцидент → уведомление админу за < 5 минут
- [ ] Дешборд uptime доступен и обновляется
- [ ] Журнал аудита фильтруем и экспортируем

## Интеграции (минимум)

- Frontend: прокинуть события ошибок в Sentry (после подключения @sentry/react)
- Backend: реализовать правила инцидентов и доставку (email/Telegram/Slack)
- UI: в Owner Dashboard — статусная карта/таблица проблем и фильтры
